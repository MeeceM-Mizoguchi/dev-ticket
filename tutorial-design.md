## 1. 目的と要件

### チケット要件
- チュートリアル機能を追加する。
- 基本的な使い方を **実際に画面を操作してレクチャー** する。

### 確定した方針（ヒアリング済み）
| 決定事項 | 内容 |
|---|---|
| **実演方式** | 対話型ガイド。スポットライトで対象要素を強調し、ユーザー自身が操作する。正しく操作されるまで次へ進まない。 |
| **舞台/データ** | **実アプリ画面**（モック/サンドボックスではなく本物の画面上でガイド）。 |
| **起動タイミング（自動）** | 初回ログイン時のみ自動起動。招待受諾→パスワード変更→ダッシュボード遷移した瞬間に起動。 |
| **未完了時の再試行** | タブ/ブラウザを閉じるなどで表示できなかった場合、次回ログイン時やダッシュボードタブ再起動時に再度表示を試みる。完了 or 明示スキップするまで「未完了」扱い。 |
| **起動タイミング（手動）** | 共通ヘッダーに新設する **ヘルプアイコン → メニュー内「チュートリアル」** ボタンからいつでも再生可能。 |

### 「自動で操作」と「対話型」の折衷
チケットは「自動で操作してレクチャー」だが、方式は対話型ガイドを選択。そこで:
- **下地（副作用のない前進）は自動送り**: ページ遷移、パネルを開く、スクロール、ハイライト移動などはツアーエンジンが自動実行。
- **核となる操作はユーザーに実行させる**: 「新規作成ボタンを押す」「ステータスを変更する」等の学習ポイントは、スポットライトで促し、ユーザーの実操作を検知して前進。

これにより「勝手に進むだけで身につかない」問題を避けつつ、体感は自動進行に近づける。

## 4. ステップエンジンのデータモデル

ツアーは「ステップ定義（データ）」＋「汎用エンジン」で構成し、フロー追加をデータ追記だけで行えるようにする。

```ts
type TutorialStep = {
  id: string;                       // ステップ識別子
  route?: string;                   // このステップで居るべきパス。違えば自動 navigate
  anchor?: TutorialAnchor;          // ハイライト対象（data-tutorial の値）。省略時は画面中央モーダル型
  placement?: "top"|"bottom"|"left"|"right"|"center";
  title: string;                    // 吹き出し見出し（日本語）
  body: string;                     // 説明（日本語）
  // --- 前進条件 ---
  advance:
    | { type: "next-button" }                    // 「次へ」クリックで進む（説明のみ）
    | { type: "user-click" }                     // 対象要素をユーザーがクリックしたら進む
    | { type: "user-action"; event: "input"|"change"|"submit" }
    | { type: "predicate"; check: () => boolean; poll?: number } // 状態を監視して進む
    | { type: "route-change"; to: string };      // 指定パスに到達したら進む
  // --- 自動下地（副作用の無い前進のみ） ---
  autoBefore?: () => void | Promise<void>;       // 例: パネルを開く/スクロール
  // --- ガード ---
  requiredPermission?: keyof UserPermissions;    // 権限が無ければステップを skip
  allowInteractionOutsideAnchor?: boolean;       // 既定 false（対象以外はクリック遮断）
};

type TutorialFlow = {
  id: string;                       // 例: "basic"
  title: string;                    // ヘルプメニュー表示名
  steps: TutorialStep[];
};
```

### エンジンの責務
1. **route 整合**: `step.route` と現在パスが違えば自動 `navigate`。到達を待ってからハイライト。
2. **アンカー解決**: `document.querySelector('[data-tutorial="<anchor>"]')` で要素取得。見つからない場合はリトライ（マウント待ち・最大 N ミリ秒）→ それでも無ければ「対象が見つからない」フォールバック（§8）。
3. **スポットライト追従**: `ResizeObserver` + スクロール/`requestAnimationFrame` で矩形を更新。対象を必要に応じて `scrollIntoView`。
4. **インタラクションゲート**: バックドロップは対象のくり抜き部分だけ pointer-events を透過。対象操作を検知したら `advance`。
5. **権限適応**: `requiredPermission` を満たさないステップは自動スキップ（§7）。
6. **永続化**: 完了/スキップ時に状態を保存（§6）。

## 6. 永続化と自動起動ロジック

### DB スキーマ追加（`supabase/add_tutorial_status.sql`）
```sql
-- 三状態: NULL/未設定 = 未実施, 'completed' = 完了, 'skipped' = スキップ
alter table profiles add column if not exists tutorial_status text;
alter table profiles add column if not exists tutorial_completed_at timestamptz;
```
- 既存の SQL 命名慣習（`supabase/add_*.sql`）に合わせる。RLS は本人のみ更新可の既存ポリシー範囲で対応。

### 状態遷移
```
未実施(null) ──自動 or 手動で起動──▶ 実行中(メモリ状態のみ)
   ▲                                   │
   │ 途中でタブ/ブラウザを閉じた                │ 完了
   └───────────────────────────┘   │ or 明示スキップ
        （DBは未実施のまま＝再試行対象）        ▼
                                completed / skipped（自動起動しない）
```
- **重要**: 「実行中」はDBに保存しない。中断＝未実施のまま。これにより「表示できなかった場合は再試行」を自然に満たす。
- 明示的な「スキップ（もう表示しない）」を選んだ時のみ `skipped` を保存。
- localStorage に完了フラグをミラーし、初回描画のちらつき（未取得中に一瞬起動）を防ぐ。

### 自動起動の発火点
- `TutorialProvider` が「ログイン済み」かつ「現在パスが `/dashboard`」になったタイミングで `tutorial_status` を確認し、未実施なら `start("basic")`。
- ネイティブ（タブ）では「ダッシュボードタブを開き直した時」も `/dashboard` へのマウントとして検知できるため同ロジックで再試行される。
- 手動起動（ヘルプメニュー）は状態に関わらず常に `start()` 可能。

## 7. シェル二系統・権限・タブへの対応（非自明な設計判断）

1. **オーバーレイのマウント位置**
   - `LinkPreviewPanel` と同様に **Provider 直下のグローバル位置**へ配置し、`AppShell`/`TabbedShell` どちらでも最前面に出す。
   - ネイティブのタブは非アクティブタブが `visibility:hidden`+`inert` でマウント維持されるため、**アンカー解決は「アクティブタブ配下」に限定**する（`inert` 要素を誤ってヒットさせない）。`TabContext` の `getActiveTabPath`/アクティブタブDOMを基準にスコープする。

2. **権限による分岐**
   - サイドバー項目・作成系ボタンは `UserPermissions`（例: `canCreateTicket`）で出し分けられる。招待直後のユーザーが `developer` 等で作成権限を持たない場合がある。
   - 各ステップに `requiredPermission` を持たせ、**満たさないステップは自動スキップ**。結果、権限に応じて「見るだけ／作成もできる」フローに自動縮退する。
   - 基本フローは「誰でも通れる最小コア（ナビ＋自分のアクション確認）」＋「権限があれば追加（チケット作成→状態変更）」の構成にする。

3. **z-index / ESC**
   - オーバーレイ z-index は既存モーダル（z-300 台）**より上**に置く。ただしツアーがダイアログ（`NewTicketDialog` 等）を対象にする場面では、対象ダイアログを隠さないようスポットライトのくり抜きで前面に見せる。
   - ESC は `escStack.push` で「中断確認（§8）」を登録。多重登録に注意（ダイアログと競合しないよう pop を確実に）。

## 8. 実データ・中断・例外時のふるまい（要検討ポイント）

「実アプリ画面」で対話操作させるため、**副作用（実データ生成）** をどう扱うかが最大の論点。以下を提案（実装前に要合意）。

- **提案（推奨）**: チケット作成ステップは、ツアー用の説明を出しつつ **ユーザーの通常操作をそのまま学習に使う**。作成されるチケットは本物。ただし、
  - 招待直後で対象プロジェクトが無い/権限が無いケースでは、当該ステップを **「作成ダイアログを開いて閉じる（保存させない）」体験に縮退**、または権限ガードで skip。
  - 「本当に保存されます」旨を吹き出しで明示し、誤操作を避ける。
- 代替案（将来）: 「チュートリアル用サンプルプロジェクト」を初回に自動用意し、そこで安全に作成体験させる（実装コスト高のため MVP では見送り）。

その他の例外:
- **対象要素が見つからない/画面構成が違う**: 一定時間リトライ→ダメなら該当ステップを skip し、`console.warn` ＋（任意で）フォールバック説明モーダル。ツアー全体は止めない。
- **ユーザーが対象外を触ろうとする**: インタラクションゲートで遮断し、吹き出しを軽く強調（シェイク等）。
- **中断（ESC / 「終了」）**: 「チュートリアルを終了しますか？（後でヘルプから再開できます）」の確認。終了＝DBは未実施のまま（再試行対象）。別途「今後表示しない」を選べば `skipped` 保存。
- **ウィンドウリサイズ/スクロール**: スポットライトと吹き出しを追従再計算。

## 9. 基本フロー詳細設計（MVP）「Dev Ticket をはじめよう」

> スコープはヒアリングで明示指定が無かったため、**まず「基本フロー1本」を MVP** とし、エンジンは複数フロー対応で作る（機能別チュートリアルは後日データ追加で拡張）。
> 対象読者は**招待直後の新規メンバー**（`developer` 等、権限が限定的な場合あり）。所要 2〜3 分。全 18 ステップ、5 セグメント。◎=コア/○=推奨/△=任意(削っても成立)。

### 前進条件の方針（対話型 × データ安全性）
- **ナビゲーション系・チケットを開く**＝ハードゲート（実操作するまで進まない）。学習効果が高く副作用が無い。
- **状態変更・コメント・作成（実データを変える操作）**＝ソフトゲート（「やってみよう」と促すが「次へ」でも進める）。実データ破壊を強制しないため。§8 の副作用方針と整合。
- **説明のみ**＝「次へ」で前進（下地の遷移・スクロールは自動送り）。

### 分岐サマリ
- **権限分岐**: `canCreateTicket` 無し → #13-14 を自動 skip。`canEditDelete` 無し/担当外 → #11 は説明のみ。
- **データ分岐**: プロジェクト0件 → #6 で説明して B〜C を skip、#15 へジャンプ。チケット0件 → #9 以降のチケット系を skip。

### 実装上の要点（フロー起因）
- **状態変更はDnDでなく詳細パネル経由で教える**（#11）。ボードのドラッグ&ドロップは対話型で誘導・検知しづらいため。
- #9→#10 は詳細パネルの**マウント/アニメ完了待ち**（predicate ポーリング）。
- #6 の route-change は `/:projectSlug` 系への遷移を検知。
- MVP を短くしたい場合は △ を落として **14 ステップ**、さらに ○ を絞って **10 ステップ**まで圧縮可能（重要度列で調整）。

## 9.5 便利機能・機能別チュートリアル・カタログ ★拡張

基本フロー（§9）は「初回に自動起動する最短の道案内」。それとは別に、**便利機能や各機能を実画面でハンズオンで覚える個別チュートリアルを多数用意**し、ヘルプメニュー（§10）から任意に起動できるようにする。すべて **§4 の同一エンジンで動くデータ（ステップ定義）** として追加するだけで増やせる。

### 棲み分け（ENHA2-024 マニュアルとの関係・§13）
- **マニュアル**＝全機能を「読んで調べる」リファレンス（8カテゴリで網羅）。
- **チュートリアル**＝実画面を「触って覚える」ハンズオン。**マニュアルと同じ8カテゴリIAに揃える**。
- 連携: マニュアル各章に「▶ 実際にやってみる」ボタン → 対応するチュートリアルを起動。チュートリアル完了画面に「詳しくはマニュアルへ」。

## 11. スコープと段階リリース

**MVP（このチケット・第1弾）**
- ツアーエンジン（overlay/spotlight/coachmark/gate/権限適応）— **全チュートリアル共通基盤**

- 自動起動（初回ダッシュボード）＋再試行
- ヘルプメニュー＋**チュートリアル・カタログUI**（§9.5 / §10）
- 二系統シェル（Web/ネイティブタブ）対応
- チュートリアル本体は **◎（優先実装）を第1弾**で搭載: 基本フロー / プロジェクト / チケット作成 / チケット詳細 / 一括作成 / マイアクション / グローバル検索 / マイフィルター（§9.5）

**第2弾以降（同エンジンにデータ追加で拡充）**
- ○ → △ の機能別チュートリアル（レビューフロー/ガント/子チケット/Wiki/議事録/メンバー招待/ロール権限 ほか、§9.5 カタログ）
- ハイブリッド化（自動実演→やってみよう）
- サンドボックス（LPモック再利用）での安全な作成体験
- i18n 対応（文言外出しは MVP から意識）

> 便利機能・全機能のカバレッジは §9.5 のカタログが担う。エンジンは1つ、各チュートリアルは**ステップ定義データを足すだけ**で増えるため、機能追加のたびに対応チュートリアルを1ファイル追加する運用にする。

## 13. 関連チケット ENHA2-024（マニュアル機能）との調整 ★重要

ENHA2-024「マニュアル機能」が**設計確定・実装未着手**で存在し、本チュートリアルと資産が重複する。**二重実装を避け、共有する**方針とする。

| 項目 | ENHA2-024（マニュアル） | 本件（チュートリアル） | 調整方針 |
|---|---|---|---|
| Topbar「?」ヘルプアイコン | ベル右に「?」→ `/manual` へ | ベル右に「?」→メニュー「チュートリアル」 | **「?」アイコンは1つに統一**。クリックで小メニューを開き「マニュアルを見る」「チュートリアルを見る」を並べる。どちらが先に実装されても片方が生えるだけにする。 |
| 強調 primitive | `<Spotlight>`（対象以外グレー＋赤枠、`boxShadow:0 0 0 9999px rgba(244,245,246,.86)`＋`border 2.5px #EF4444`、variant=`look`/`click`、`ClickHotspot` 波紋） | スポットライト＋くり抜き | **ENHA2-024 の `<Spotlight>` を再利用/共通化**。ただし ENHA2-024 版はモック内 rect 前提。本件は**実DOM要素の rect**を渡すため、rect 取得部（`getBoundingClientRect`＋追従）を上に被せる。primitive の見た目（グレー＋赤枠＋波紋）は流用。 |
| 舞台 | 専用モック＋LP資産、`/manual` ルート（分量大でモーダル不可） | 実アプリ画面（ツアー） | 別物。マニュアル=読み物、チュートリアル=実操作、と役割分担。相互リンク（チュートリアル完了画面から「詳しくはマニュアル」へ）。 |
| スタイル規約 | 保護下ページは**インラインスタイル**（Dashboard は className 0）。LP の Tailwind は持ち込まない。 | 同上を踏襲 | **本設計もインラインスタイルで実装**（§3 の CoachMark 等）。Tailwind は使わない。 |

> 実装順の推奨: 先に「?」アイコン＋共有メニューの器と `<Spotlight>` primitive を（どちらのチケットで作るにせよ）共通化しておくと、両機能がきれいに乗る。
